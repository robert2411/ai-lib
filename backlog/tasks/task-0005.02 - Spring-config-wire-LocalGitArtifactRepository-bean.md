## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
1. Create SecurityConfig (src/main/java/com/agentlibrary/config/SecurityConfig.java)
   - @Configuration @EnableWebSecurity
   - @Bean SecurityFilterChain filterChain(HttpSecurity http):
     * Disable CSRF (API-first, stateless for CLI clients)
     * Configure HTTP Basic authentication (for API/CLI)
     * Configure form login with /login page (for browser UI — future milestone)
     * Session creation policy: IF_REQUIRED (browser sessions OK)
     * Authorization rules:
       - GET /api/v1/** → authenticated (any role)
       - POST|PUT|DELETE /api/v1/artifacts/** → hasAnyRole("EDITOR", "ADMIN")
       - POST /api/v1/admin/** → hasRole("ADMIN")
       - GET /healthz → permitAll
       - All other requests → authenticated
   - @Bean UserDetailsService userDetailsService(AppProperties props):
     * Load users from users.yaml file (path from props.getUsersFile())
     * Parse YAML: list of {username, passwordHash, roles}
     * Return InMemoryUserDetailsManager with parsed users
     * Passwords are bcrypt-encoded in the YAML file
   - @Bean PasswordEncoder passwordEncoder():
     * Return BCryptPasswordEncoder

2. Create RateLimitConfig (src/main/java/com/agentlibrary/config/RateLimitConfig.java)
   - @Configuration
   - @Bean Bucket loginRateLimitBucket():
     * Configure bucket4j bucket: 5 attempts per minute, refill greedy
     * Used by login endpoint to throttle brute-force attempts
   - Note: actual filter/interceptor integration is lightweight — just expose the Bucket bean
     for use in a RateLimitFilter or login handler

3. Create RateLimitFilter (src/main/java/com/agentlibrary/config/RateLimitFilter.java)
   - Extends OncePerRequestFilter
   - Inject loginRateLimitBucket
   - Only applies to POST /login
   - If bucket.tryConsume(1) fails, return 429 Too Many Requests
   - Otherwise chain.doFilter()

4. Update application.yml (src/main/resources/application.yml)
   - Existing config is sufficient for dev profile
   - library.data-dir, repo-path, work-path, users-file already defined

5. Create application-prod.yml (src/main/resources/application-prod.yml) (AC #2)
   - Override paths for container deployment:
     library:
       data-dir: /data
       repo-path: /data/library.git
       work-path: /data/library-work
       users-file: /data/users.yaml
   - server.port: 8080

6. Verify bean wiring (AC #1) — all components already use @Service/@Configuration:
   - StorageConfig already wires: WriteQueue, ArtifactRepository (LocalGitArtifactRepository)
   - IndexConfig already wires: IndexService ↔ SearchService ↔ Repository callback
   - FacetService is @Service with IndexService injection
   - ArtifactService (from TASK-0005.01) is @Service with constructor injection
   - SecurityConfig wires UserDetailsService, PasswordEncoder
   - RateLimitConfig wires Bucket
   - No additional manual bean definitions needed — Spring component scanning handles it

7. Create sample users.yaml for dev/test (data/users.yaml or src/test/resources/users.yaml)
   - Format:
     users:
       - username: admin
         password: $2a$10$... (bcrypt hash)
         roles: [ADMIN, EDITOR, USER]
       - username: editor
         password: $2a$10$...
         roles: [EDITOR, USER]
       - username: reader
         password: $2a$10$...
         roles: [USER]

8. Integration test (src/test/java/com/agentlibrary/config/SecurityConfigTest.java)
   - @SpringBootTest @AutoConfigureMockMvc
   - Test context loads successfully (AC #1)
   - Test unauthenticated GET /api/v1/artifacts returns 401
   - Test authenticated GET /api/v1/artifacts with USER role returns 200
   - Test POST /api/v1/artifacts with USER role returns 403
   - Test POST /api/v1/artifacts with EDITOR role returns allowed
   - Test POST /api/v1/admin/reindex with EDITOR returns 403
   - Test POST /api/v1/admin/reindex with ADMIN returns allowed

9. Dependencies: Requires TASK-0005.01 (ArtifactService must exist as a bean)
