## Implementation Plan — TASK-0006.02: FileUserDetailsService and Spring Security Wiring

### Context
SecurityConfig already has:
- `filterChain` with form login at /login, HTTP Basic, CSRF disabled, role-based matchers (ADMIN, EDITOR)
- `userDetailsService` bean that directly parses YAML (to be replaced)
- `passwordEncoder` bean (BCryptPasswordEncoder)
- Roles in use: ADMIN, EDITOR (SecurityConfig matchers), USER (users.yaml)

The task description mentions ADMIN > EDITOR > VIEWER hierarchy. However, existing code uses USER not VIEWER. The design doc says "Roles: USER, ADMIN". SecurityConfig uses EDITOR too. We will use the roles already in code (ADMIN, EDITOR, USER) and define hierarchy: ADMIN > EDITOR > USER.

### Steps

1. **Create `com.agentlibrary.auth.FileUserDetailsService`**
   - File: `src/main/java/com/agentlibrary/auth/FileUserDetailsService.java`
   - Implements `org.springframework.security.core.userdetails.UserDetailsService`
   - Annotate with `@Service`
   - Constructor-inject `UsersFile` (from TASK-0006.01)
   - In constructor or `@PostConstruct`, call `usersFile.loadAll()` and cache results in a `Map<String, UserEntry>`
   - `loadUserByUsername(String username)`: look up in map, return Spring `User` (o.s.s.core.userdetails.User) with username, encodedPassword, authorities from roles. Throw `UsernameNotFoundException` if not found.

2. **Refactor SecurityConfig — remove inline YAML loading**
   - Remove the private `loadUsersFromYaml(Path)` method entirely
   - Remove the `userDetailsService(AppProperties)` bean method
   - The `FileUserDetailsService` @Service will be auto-detected by Spring Security (it implements UserDetailsService)
   - Keep `passwordEncoder()` bean in SecurityConfig
   - Keep `filterChain()` bean in SecurityConfig
   - Remove unused imports (SnakeYAML, Files, InputStream, etc.)

3. **Add RoleHierarchy bean to SecurityConfig**
   - Add `@Bean RoleHierarchyImpl roleHierarchy()` method
   - Define: `ROLE_ADMIN > ROLE_EDITOR > ROLE_USER`
   - This enables ADMIN to access EDITOR-only and USER-only endpoints automatically
   - Register via `@Bean static GrantedAuthorityDefaults` or ensure `RoleHierarchy` is picked up by the `HttpSecurity` config
   - Actually, Spring Security 6.x picks up RoleHierarchy bean automatically when using `@EnableMethodSecurity` + authorizeHttpRequests if you register it properly. Wire into filterChain or use `expressionHandler`.

4. **Add logout support to filterChain**
   - The existing config has `.formLogin(form -> form.loginPage("/login").permitAll())`
   - Add `.logout(logout -> logout.logoutUrl("/logout").logoutSuccessUrl("/login?logout").permitAll())`
   - Ensure /logout POST is permitted

5. **Update SecurityConfigTest**
   - File: `src/test/java/com/agentlibrary/config/SecurityConfigTest.java`
   - Test valid credentials authenticate (use @WithMockUser or HTTP Basic with test user)
   - Test invalid credentials return 401 for API / redirect for browser
   - Test ROLE_ADMIN can access admin endpoints (POST /api/v1/admin/reindex)
   - Test ROLE_USER (VIEWER equivalent) cannot POST/PUT/DELETE artifacts
   - Test role hierarchy: ADMIN inherits EDITOR access

6. **Update SecurityConfigYamlParseTest**
   - These tests directly instantiate SecurityConfig and call `userDetailsService()`. After refactoring, they should test UsersFile directly (which is already covered by TASK-0006.01 tests).
   - Either: delete this test class (covered by UsersFileTest), or refactor to test FileUserDetailsService integration
   - Recommendation: delete and rely on UsersFileTest + SecurityConfigTest integration tests

7. **Create FileUserDetailsService unit test**
   - File: `src/test/java/com/agentlibrary/auth/FileUserDetailsServiceTest.java`
   - Test: valid user returns correct UserDetails with expected authorities
   - Test: unknown username throws UsernameNotFoundException
   - Test: roles map correctly to Spring GrantedAuthority (ROLE_ prefix)
   - Mock or provide a real UsersFile with test data
