## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
BcryptCli is a small main class that accepts a plaintext password arg and prints a BCrypt hash. scripts/gen-password-hash.sh wraps it: runs the JAR or mvn exec:java, prints the hash for pasting into users.yml.

## Implementation Plan — TASK-0006.04: BcryptCli and gen-password-hash.sh

### Context
No scripts/ directory exists yet. The design doc specifies: "Ship scripts/gen-password-hash.sh to generate bcrypt hashes without running the full app." BcryptCli is a standalone main class that can be run via `mvn exec:java` or directly from the JAR.

### Steps

1. **Create `com.agentlibrary.cli.BcryptCli` class**
   - File: `src/main/java/com/agentlibrary/cli/BcryptCli.java`
   - Contains `public static void main(String[] args)`
   - Accepts exactly one argument: the plaintext password
   - If no args or too many args: print usage to stderr and exit(1)
   - Usage: `Usage: BcryptCli <password>`
   - Uses `org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder` directly (no Spring context needed)
   - Instantiate `new BCryptPasswordEncoder()` (default strength 10)
   - Print `encoder.encode(password)` to stdout (just the hash, no trailing newline beyond println)
   - Exit 0 on success

2. **Create `scripts/` directory and `gen-password-hash.sh`**
   - File: `scripts/gen-password-hash.sh`
   - Make executable (chmod +x noted in script header)
   - Shebang: `#!/usr/bin/env bash`
   - Accept password as first argument; if missing, prompt interactively (read -s -p)
   - Two execution strategies (try in order):
     a. If `target/ai-library-*.jar` exists: `java -cp target/ai-library-*.jar com.agentlibrary.cli.BcryptCli "$password"`
     b. Otherwise: `./mvnw -q exec:java -Dexec.mainClass=com.agentlibrary.cli.BcryptCli -Dexec.args="$password"`
   - Print only the hash to stdout (all other output to stderr)
   - Exit code mirrors the java process exit code
   - Add header comment explaining purpose and usage

3. **Add exec-maven-plugin configuration to pom.xml (optional)**
   - The `mvn exec:java` approach works without explicit plugin config if the class is on classpath
   - However, adding `exec-maven-plugin` in `<pluginManagement>` makes it cleaner
   - This is optional — `mvnw exec:java -Dexec.mainClass=...` works with just the dependency on classpath
   - Decision: skip plugin config unless mvn exec:java fails without it

4. **Write unit test for BcryptCli**
   - File: `src/test/java/com/agentlibrary/cli/BcryptCliTest.java`
   - Test: calling main with a password prints a valid BCrypt hash (starts with $2a$ or $2b$)
   - Test: the printed hash verifies against BCryptPasswordEncoder.matches()
   - Test: calling main with no args exits with non-zero (or throws / prints usage)
   - For stdout capture: redirect System.out to ByteArrayOutputStream in test setup
   - Alternatively, extract logic into a testable method and test that directly

5. **Verify end-to-end**
   - Integration verification (manual or scripted):
   - Run: `./scripts/gen-password-hash.sh testpassword`
   - Take output hash, put in users.yaml
   - Boot app, authenticate with testpassword → success
   - This validates AC#2 (hash verifies against Spring BCryptPasswordEncoder)

### AC Mapping
- AC#1 (gen-password-hash.sh prints valid BCrypt hash) → Steps 2 + 1, verified by test + script execution
- AC#2 (hash verifies against Spring BCryptPasswordEncoder) → Step 1 (uses same encoder), verified by unit test

### Dependencies
- Depends on: None (standalone utility, only needs spring-security-crypto on classpath which is already present)
- No downstream dependencies (but users.yaml entries will use hashes generated by this tool)

### Notes
- BCryptPasswordEncoder is in spring-security-crypto which is transitively included via spring-boot-starter-security
- The JAR classpath approach requires the project to be built first (mvn package). The mvnw fallback handles the case where JAR doesn't exist yet.
- Script should work on macOS and Linux (bash 3.2+, per design doc runtime deps)

## Implementation Notes

Self-review complete. Plan covers all AC. No gaps or unverified assumptions. BCryptPasswordEncoder is available on classpath (confirmed via spring-boot-starter-security in pom.xml). Script follows bash 3.2+ compatibility per design doc. No blockers. Analysis complete. Plan ready.

🔍 PLAN REVIEW CONCERNS:
- Concern #1: Script strategy (a) "java -cp target/ai-library-*.jar com.agentlibrary.cli.BcryptCli" WILL NOT WORK. Spring Boot 3.3.4 with spring-boot-maven-plugin produces a fat JAR where classes live under BOOT-INF/classes/, not at the JAR root. Using -cp on a fat JAR gives ClassNotFoundException. The script will always fail on strategy (a) when the JAR exists, likely exiting with an error instead of falling through to strategy (b).
- Concern #2: The plan says "try in order" but does not specify the fallthrough mechanism. If the script uses an if-file-exists condition (which the wording implies), strategy (b) is unreachable after a build because the JAR file WILL exist. Clarify: should strategy (a) be removed entirely, or should the script catch failure from (a) and retry with (b)?
- Recommended fix: Use ONLY "./mvnw -q exec:java" as the execution strategy (always works without a build step), OR use "java -cp target/classes -Dloader.path=... org.springframework.boot.loader.launch.PropertiesLauncher" which requires mvn compile but not package, OR extract a thin JAR via maven-jar-plugin with a separate classifier.

Verdict: Plan needs revision for Step 2 (script execution strategy) before implementation. Steps 1, 3, 4, 5 are solid. AC mapping is correct conceptually but the script will fail in practice.
