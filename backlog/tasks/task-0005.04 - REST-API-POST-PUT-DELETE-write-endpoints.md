## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
POST /api/v1/artifacts (create). PUT /api/v1/artifacts/{slug}/{version} (update). DELETE /api/v1/artifacts/{slug}/{version}. All write endpoints require ROLE_EDITOR or ROLE_ADMIN. Request body is raw artifact content (multipart or text/plain).
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [x] #1 POST creates artifact and returns 201 with location header
- [x] #2 PUT updates artifact content
- [x] #3 DELETE removes artifact version, returns 204
- [x] #4 Unauthenticated requests to write endpoints return 401

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
1. Create ArtifactCreateRequest DTO (src/main/java/com/agentlibrary/web/api/ArtifactCreateRequest.java)
   - Java record: String content (raw artifact with YAML frontmatter)
   - Content-Type: text/plain or application/json with content field
   - Alternatively: accept raw body as @RequestBody String (simpler for CLI clients sending plain text)

2. Add write endpoints to ArtifactApiController (src/main/java/com/agentlibrary/web/api/ArtifactApiController.java)
   - All write endpoints secured with @PreAuthorize("hasAnyRole('EDITOR', 'ADMIN')") or rely on SecurityConfig URL rules
   - Inject principal from SecurityContextHolder for User construction

3. Implement POST /api/v1/artifacts — UPSERT semantics (AC #1)
   - Method: create(@RequestBody String rawContent, Principal principal)
   - Extract User from principal (username + roles from SecurityContext)
   - Call artifactService.createOrUpdate(rawContent, user)
   - IMPORTANT: POST is an UPSERT (createOrUpdate). If slug already exists, update it.
     No 409 Conflict needed. This matches the service layer which calls createOrUpdate().
     Rationale: simplifies CLI workflow — clients can POST without checking existence first.
   - Return 201 Created with:
     * Location header: /api/v1/artifacts/{slug}/{version}
     * Body: ArtifactResponse of the created/updated artifact
   - ValidationException → 422 (handled by ApiExceptionHandler)

4. Implement PUT /api/v1/artifacts/{slug}/{version} (AC #2)
   - Method: update(@PathVariable String slug, @PathVariable String version, @RequestBody String rawContent, Principal principal)
   - Verify slug in path matches slug in frontmatter content (reject mismatches with 400)
   - Extract User from principal
   - Call artifactService.createOrUpdate(rawContent, user)
   - Return 200 OK with updated ArtifactResponse
   - NotFoundException → 404, ValidationException → 422

5. Implement DELETE /api/v1/artifacts/{slug} (AC #3)
   - Path: DELETE /api/v1/artifacts/{slug} — NO {version} path variable
   - Rationale: ArtifactRepository.delete(type, name, message, user) removes the entire
     artifact folder (all versions). The design doc confirms: "Delete: commit removes folder."
     There is no version-specific deletion in the repository contract.
   - Method: delete(@PathVariable String slug, Principal principal)
   - Extract User from principal
   - Call artifactService.delete(slug, user)
   - Return 204 No Content (AC #3)
   - NotFoundException → 404

6. Helper method: extractUser(Principal principal) -> User
   - Get Authentication from SecurityContextHolder
   - Extract username and authorities (roles)
   - Construct User record
   - Used by all write endpoints

7. Handle authentication enforcement (AC #4)
   - SecurityConfig (TASK-0005.02) already restricts POST/PUT/DELETE /api/v1/artifacts/** to EDITOR/ADMIN
   - Unauthenticated requests get 401 from Spring Security before reaching controller
   - Insufficient role gets 403

8. Unit tests (src/test/java/com/agentlibrary/web/api/ArtifactWriteEndpointsTest.java)
   - @WebMvcTest(ArtifactApiController.class)
   - @MockBean ArtifactService

   Test POST /api/v1/artifacts — upsert (AC #1):
   - @WithMockUser(roles="EDITOR") + valid raw content body
   - Verify 201 status, Location header present, body contains artifact JSON
   - Verify artifactService.createOrUpdate() called with correct content
   - Test POST with slug that already exists: still returns 201 (upsert, no 409)

   Test PUT /api/v1/artifacts/{slug}/{version} (AC #2):
   - @WithMockUser(roles="EDITOR") + updated content body
   - Verify 200 status, body has updated artifact
   - Test slug mismatch between path and body returns 400

   Test DELETE /api/v1/artifacts/{slug} (AC #3):
   - @WithMockUser(roles="ADMIN") + slug path
   - Verify 204 status, no body
   - Verify artifactService.delete() called with slug (no version param)

   Test unauthenticated access (AC #4):
   - No @WithMockUser — POST returns 401
   - No @WithMockUser — PUT returns 401
   - No @WithMockUser — DELETE returns 401

   Test insufficient role:
   - @WithMockUser(roles="USER") — POST returns 403
   - @WithMockUser(roles="USER") — PUT returns 403
   - @WithMockUser(roles="USER") — DELETE returns 403

9. Dependencies: Requires TASK-0005.01 (ArtifactService), TASK-0005.02 (SecurityConfig for role enforcement)

🔍 PLAN REVIEW CONCERNS:
- Concern #1: POST semantics ambiguous. Step 3 calls artifactService.createOrUpdate() which succeeds whether artifact exists or not. But AC #1 says "POST creates artifact" — standard REST semantics expect POST to fail with 409 Conflict if the resource already exists. The plan must either: (a) add a check in the controller: if slug already in index, return 409; or (b) explicitly document that POST is create-or-update (upsert) and update the AC wording.
- Concern #2: DELETE path is /api/v1/artifacts/{slug}/{version} but ArtifactRepository.delete(String type, String name, String message, User user) has NO version parameter — it removes the entire artifact folder. The {version} path variable is captured but never passed to the service/repo. Either: (a) remove {version} from the DELETE path (make it DELETE /api/v1/artifacts/{slug}); or (b) implement version-specific deletion in the service (which would require a new repo method). The design doc says DELETE removes the folder (all versions), so option (a) aligns with the spec.

Verdict: Plan needs revision before implementation — two contract inconsistencies must be resolved.

Plan updated per review: (1) POST semantics clarified as UPSERT — calls createOrUpdate(), no 409 Conflict. If slug exists, update it. Matches service layer contract. Added upsert test case. (2) DELETE path fixed from /api/v1/artifacts/{slug}/{version} to /api/v1/artifacts/{slug} — no {version} param. ArtifactRepository.delete() removes entire folder (all versions) per design doc. Method signature updated accordingly.
