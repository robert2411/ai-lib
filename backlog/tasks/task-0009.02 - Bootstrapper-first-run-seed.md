7. Create Bootstrapper component: src/main/java/com/agentlibrary/bootstrap/Bootstrapper.java
   - @Component, implements ApplicationListener<ApplicationReadyEvent>
   - Inject: ArtifactService, IndexService, AppProperties
   - onApplicationEvent():
     a. Check if repo is empty: indexService.getAll().isEmpty()
     b. If empty → call seedAll()
     c. If not empty → log "Repo already seeded, skipping bootstrap" and return
   - Maps to AC #2 (idempotency — subsequent startups skip)

8. Implement seedAll() method in Bootstrapper with per-seed error isolation
   - Define list of seed resource paths: List.of("seeds/agent-library-install.md", "seeds/agent-lib-sh.md", ...)
   - For each resource path:
     a. Read classpath resource via getClass().getResourceAsStream("/" + path)
     b. Read full content as UTF-8 string
     c. try {
          artifactService.createOrUpdate(content, SYSTEM_USER);
          log.info("Seeded artifact: {}", slug);
        } catch (Exception e) {
          log.warn("Failed to seed {}: {} — continuing with remaining seeds", path, e.getMessage());
          // Do NOT rethrow — continue processing remaining seed files
        }
   - SYSTEM_USER: new User("system", Set.of("ADMIN"))
   - Order: seed individual artifacts BEFORE agent-group (to maximise chance group members exist)
   - Per-seed error isolation ensures one bad seed file does not abort entire bootstrap

9. Handle group validation — use ArtifactService.createOrUpdate() with validator support
   - DECISION (single approach): Use ArtifactService.createOrUpdate() for ALL seeds including the group
   - The MetadataValidator should accept type=agent-group. Check MetadataValidator code:
     * If it already handles agent-group type → no changes needed
     * If it rejects agent-group → add agent-group to the allowed types list in MetadataValidator
   - Group member validation (checking member slugs exist): the per-seed try-catch from Step 8
     will catch any ValidationException if member slugs don't exist, log a warning, and continue
   - This means dev-team-group will still be committed if validation is lenient, OR will be
     skipped with a warning if strict — either outcome is acceptable for an example seed
   - No new methods, no unchecked bypass, no direct repository.save() calls
   - Keeps Bootstrapper using the standard service API path

10. Ensure search indexes are rebuilt after seeding
    - After all seeds committed, trigger indexService.refresh() explicitly
    - Or rely on repository commit callback (already wired via IndexConfig)
    - Verify: each save() via ArtifactService triggers refreshCallback → IndexService.refresh() → SearchService.rebuild()
    - The existing wiring should handle this: save → commit → callback → refresh
    - After seeding, artifacts should be immediately searchable
    - Maps to AC #3

11. Spring bean ordering considerations
    - Bootstrapper uses ApplicationReadyEvent (fires after ALL beans initialised and CommandLineRunners complete)
    - This means IndexConfig's CommandLineRunner (which wires refresh callback) runs BEFORE Bootstrapper
    - So by the time Bootstrapper seeds, the callback is already wired → index will auto-refresh
    - Confirm ordering: @PostConstruct (repo init, index init) → CommandLineRunner (wire callback) → ApplicationReadyEvent (bootstrap seed)

    - Maps to AC #1, #2, #3, #4
