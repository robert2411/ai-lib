## Implementation Plan â€” TASK-0006.03: LoginRateLimiter with Bucket4j

### Context
Existing code:
- `RateLimitConfig` creates a SINGLE global `Bucket` (5 req/min) â€” NOT per-IP
- `RateLimitFilter` applies that single bucket to POST /login â€” all users share one bucket
- Task requires: per-IP buckets (10 attempts / 10 minutes), independent per IP, reset on success

The existing implementation is a skeleton that needs to be replaced with proper per-IP tracking.

### Steps

1. **Refactor `RateLimitConfig` â†’ per-IP bucket management**
   - File: `src/main/java/com/agentlibrary/config/RateLimitConfig.java`
   - Remove the single `Bucket` bean
   - Instead, provide a `LoginRateLimiter` bean (or rename the class)
   - Alternatively, create a new class `com.agentlibrary.auth.LoginRateLimiter`
   - Use a `ConcurrentHashMap<String, Bucket>` to store per-IP buckets
   - Method: `Bucket resolveBucket(String ipAddress)` â€” returns or creates bucket for that IP
   - Bucket config: capacity=10, refillGreedy(10, Duration.ofMinutes(10))
   - Method: `void resetBucket(String ipAddress)` â€” removes entry from map (resets on success)
   - Consider: add eviction for stale IPs (optional for v1, note it as future improvement)

2. **Refactor `RateLimitFilter` to use per-IP buckets**
   - File: `src/main/java/com/agentlibrary/config/RateLimitFilter.java`
   - Inject `LoginRateLimiter` instead of `Bucket`
   - Extract client IP from request: `request.getRemoteAddr()` (also check X-Forwarded-For header since app runs behind reverse proxy)
   - Call `loginRateLimiter.resolveBucket(clientIp).tryConsume(1)`
   - If exhausted: return 429 with "Too Many Requests" body
   - Add IP extraction helper method that handles proxy headers

3. **Implement successful-login reset**
   - Option A: Use a Spring Security `AuthenticationSuccessHandler` that calls `loginRateLimiter.resetBucket(ip)`
   - Option B: Use a custom `AuthenticationSuccessListener` (ApplicationListener<AuthenticationSuccessEvent>)
   - Option A is preferred â€” wire into SecurityConfig formLogin: `.successHandler(customHandler)`
   - Create `LoginSuccessHandler` that delegates to default behaviour AND resets bucket
   - Wire in SecurityConfig: `.formLogin(form -> form.loginPage("/login").successHandler(loginSuccessHandler).permitAll())`
   - Also handle HTTP Basic success: use `AuthenticationSuccessEvent` listener for comprehensive coverage

4. **Register filter in correct position in chain**
   - Current approach: `@Component` + `OncePerRequestFilter` auto-registers
   - Better: explicitly add before `UsernamePasswordAuthenticationFilter` in SecurityConfig filterChain
   - Use `http.addFilterBefore(rateLimitFilter, UsernamePasswordAuthenticationFilter.class)`
   - Remove `@Component` from RateLimitFilter to avoid double-registration
   - OR keep @Component and use FilterRegistrationBean to disable auto-registration + add manually

5. **Write unit tests for LoginRateLimiter**
   - File: `src/test/java/com/agentlibrary/auth/LoginRateLimiterTest.java` (or config package)
   - Test: first 10 attempts from same IP succeed (tryConsume returns true)
   - Test: 11th attempt returns false (bucket exhausted)
   - Test: different IPs have independent buckets
   - Test: `resetBucket(ip)` allows new attempts from that IP
   - Test: after refill period, bucket replenishes (use Bucket4j test clock if available, or just verify bucket creation params)

6. **Write integration test for rate limiting**
   - File: `src/test/java/com/agentlibrary/config/RateLimitFilterTest.java`
   - Test: 10 POSTs to /login from same IP â†’ all return login page/401 (not 429)
   - Test: 11th POST â†’ returns 429
   - Test: successful auth resets counter
   - Test: different IPs (via X-Forwarded-For) don't interfere
   - Use MockMvc with Spring Boot test context
