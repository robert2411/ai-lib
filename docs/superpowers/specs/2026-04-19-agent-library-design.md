# Agent Library Server — Design

**Date:** 2026-04-19
**Status:** Approved, ready for planning

## Purpose

A library server for agent artifacts — Claude skills, Claude/Copilot agents, MCP server configs, slash commands, hooks, `CLAUDE.md` snippets, OpenCode configs, prompts. A small team (<10 users) uploads, browses, and installs these artifacts into their local harnesses through a Claude skill + companion CLI.

## Scope

**First release:**

- Team auth (basic, file-based users, <10 known users)
- Local bare git repo as storage backend (abstraction allows remote-git later)
- Typed artifacts, each in its own folder in the git repo
- Tagging: harness list + free-form tags + structured facets (category, language, author, visibility)
- Explicit semver per artifact, enforced, materialised as git tags
- Full CRUD web UI (browse, upload, edit in-browser, delete, manage tags)
- Install skill + shell CLI, supporting auto-place and fetch-only modes
- Git-only metadata (YAML frontmatter / sidecar + generated `INDEX.yaml`), no DB
- Docker single-image deployment

**Explicitly out of first release:**

- Public/unauthenticated access
- Remote git backend (planned phase C)
- OIDC / 2FA
- Per-artifact ACLs
- External search service

## Architecture

Single Spring Boot 3 application. Thymeleaf server-rendered views + HTMX for partial swaps. JGit for git storage. In-memory Lucene (RAMDirectory) for search. Runs in one container; persistent volume holds the bare repo, working checkout, users file, and config overrides.

```
┌─ Browser (HTMX) ──────────┐
│  Thymeleaf views          │
└──────────┬────────────────┘
           │ HTTP
┌──────────┴──────────────────┐
│  Spring Boot                │
│  ├─ Controllers (web+API)   │
│  ├─ ArtifactService         │
│  ├─ IndexService (in-mem)   │
│  ├─ InstallService          │
│  └─ ArtifactRepository ◄────┼── JGit bare repo
└─────────────────────────────┘
           │
    /data/library.git  (volume)
```

Storage abstracted behind `ArtifactRepository` interface. `LocalGitArtifactRepository` is the first-release impl. A future `RemoteGitArtifactRepository` (phase C) clones and pushes to an external remote — no contract change expected.

## Git Repository Layout

Bare repo on volume. Working checkout separate. One folder per artifact type; one folder per artifact inside.

```
library.git/
  skills/<name>/
    skill.md          # content + YAML frontmatter
    files/…           # optional bundled assets
  agents-claude/<name>/
    agent.md
  agents-copilot/<name>/
    agent.md
  mcp/<name>/
    mcp.json          # MCP server config fragment
    meta.yaml         # sidecar metadata (JSON has no frontmatter)
  commands/<name>/
    command.md
  hooks/<name>/
    hook.json
    meta.yaml
  claude-md/<name>/
    snippet.md
  opencode/<name>/
    config.yaml
    meta.yaml
  prompts/<name>/
    prompt.md
  INDEX.yaml          # auto-generated on each commit
```

Artifact identity: `(type, name)`. `name` is a URL-safe slug and matches the folder name. Metadata lives in YAML frontmatter when the primary file is markdown; otherwise in a `meta.yaml` sidecar.

Version tags: lightweight git tags of the form `<type>/<name>@<semver>`, e.g. `skills/git-helper@1.2.0`.

## Artifact Metadata Schema

Common fields (frontmatter or `meta.yaml`):

```yaml
name: git-helper              # slug, matches folder
title: Git Helper             # display name
type: skill                   # skill|agent-claude|agent-copilot|mcp|command|hook|claude-md|opencode|prompt
version: 1.2.0                # semver
description: One-line summary
harnesses: [claude, copilot]  # which harnesses this works with
tags: [git, workflow]         # free-form
category: developer-tools     # structured facet
language: en                  # structured facet
author: robert
visibility: team              # team|private (future-proof)
created: 2026-04-19T10:00:00Z
updated: 2026-04-19T10:00:00Z
install:                      # per-type install hints
  target: ~/.claude/skills/{name}/
  files: [skill.md, files/**]
  merge: null                 # for MCP/hooks: json-merge strategy
```

Per-type extensions allowed (e.g. MCP adds `serverName`, hook adds `event`).

`INDEX.yaml` is a flat list of all artifact metadata, regenerated by the server on every commit. It drives search/browse and is fully rebuildable from repo state via `POST /admin/reindex`.

## Storage Abstraction

```java
interface ArtifactRepository {
  List<Artifact> list(Filter f);
  Optional<Artifact> get(String type, String name, String ref);  // ref = tag|sha|HEAD
  CommitResult save(Artifact a, String message, User u);         // create/update + commit + tag
  void delete(String type, String name, String message, User u);
  List<VersionRef> versions(String type, String name);
  InputStream bundle(String type, String name, String ref);      // tar.gz for install
}
```

### Write flow

1. Validate metadata against type schema.
2. Stage files in a temp tree.
3. Commit with message `"<type>/<name>: <action> by <user>"`. Author = real user (Spring Security principal), committer = server identity.
4. If `version` bumped, create a lightweight tag `<type>/<name>@<version>`.
5. Regenerate `INDEX.yaml` and commit alongside.
6. Notify `IndexService` to refresh in-memory cache.

All writes are serialised through a single-writer queue to avoid concurrent commit races.

## REST API

Base path: `/api/v1`.

```
GET    /artifacts                               list; filter: type, harness, tag, q
GET    /artifacts/{type}/{name}                 latest metadata + content
GET    /artifacts/{type}/{name}/versions
GET    /artifacts/{type}/{name}@{ver}           specific version
POST   /artifacts/{type}                        create (multipart or JSON+files)
PUT    /artifacts/{type}/{name}                 update (new commit; bump if version changed)
DELETE /artifacts/{type}/{name}                 remove folder via commit

GET    /artifacts/{type}/{name}/bundle?ref=…                    tar.gz
GET    /artifacts/{type}/{name}/install-manifest?harness=claude JSON: {targetPath, files[], mergeStrategy}

GET    /tags
GET    /harnesses
GET    /types

POST   /admin/reindex
GET    /healthz
```

Auth: HTTP Basic for API/CLI, session cookie for browser. Every call (including reads) requires auth — this is a team-only instance, not public.

## Web UI

Pages:

- `/` — dashboard: recent commits, counts per type, global search.
- `/browse` — artifact list/grid. Sidebar filters: type, harness, tag, category, language, author. HTMX swaps result grid.
- `/a/{type}/{name}` — detail view: rendered content, metadata panel, version dropdown, install snippet (copy-to-clipboard), edit button, version history.
- `/new` — type picker → dynamic form (fields per type schema) + file upload/textarea/Monaco editor.
- `/edit/{type}/{name}` — prefilled form.
- `/admin` — reindex, user list (file-backed), repo stats.
- `/login` — Spring Security form login.

HTMX usage: debounced live search, filter chips, tag autocomplete, inline save with optimistic feedback, on-demand version diff viewer. Fragments served via Thymeleaf `th:fragment`. Monaco loaded as static asset — no frontend build step.

## Auth

First release: Spring Security + file-backed users.

`/data/users.yaml`:

```yaml
users:
  - username: robert
    passwordHash: "$2a$10$…"      # bcrypt
    roles: [ADMIN]
  - username: alice
    passwordHash: "…"
    roles: [USER]
```

- Roles: `USER` (read + write) and `ADMIN` (delete, reindex, user management).
- Session cookie for browser; HTTP Basic for API/CLI.
- `/admin/users` UI to add/reset passwords (admin only). Bcrypt on write.
- No self-signup. Users provisioned manually.
- Rate-limit `/login` via bucket4j to deter brute force.
- No 2FA, no OIDC in v1. Deploy behind TLS reverse proxy (Caddy/Traefik); documented but not bundled.

Ship a helper shell script `scripts/gen-password-hash.sh` to generate bcrypt hashes without running the full app (e.g. `docker exec … gen-password-hash.sh mysecret`).

Future (not in v1): OIDC, per-artifact ACLs, separate audit log.

## Install Skill + CLI

Library hosts its own install skill — bootstrap commit seeds it.

**Layout in library repo:**

```
skills/agent-library-install/
  skill.md                  # Claude skill markdown
  scripts/
    agent-lib.sh            # POSIX shell CLI
    agent-lib-common.sh
  README.md
```

**CLI surface:**

```
agent-lib search <query> [--type T] [--harness H] [--tag T]
agent-lib show <type>/<name>[@ver]
agent-lib install <type>/<name>[@ver] [--harness H] [--fetch-only] [--dest DIR]
agent-lib list-versions <type>/<name>
agent-lib login                         # writes ~/.config/agent-lib/creds (mode 600)
agent-lib config set server <url>
```

**Install flow:**

1. `GET /install-manifest?harness=…` → `{targetPath, files[], mergeStrategy}`.
2. `GET /bundle` → tar.gz.
3. Resolve `targetPath` with harness defaults:
   - claude skill → `~/.claude/skills/<name>/`
   - claude agent → `.claude/agents/<name>.md` (cwd)
   - copilot agent → `.github/agents/<name>.md` (cwd)
   - MCP → merge into `~/.claude/settings.json` at `mcpServers.<name>` (configurable path)
   - hook → merge into `settings.json` at `hooks.<event>[]`
   - claude-md → append to nearest `CLAUDE.md`
4. `--fetch-only` skips placement — writes to `./artifact/`.
5. Dry-run mode prints planned actions.

**Skill markdown** tells Claude: parse intent (search / install / show), shell out to `agent-lib.sh`, show results, confirm destructive merges before executing them.

Runtime deps: `curl`, `jq`, `tar`, Bash 3.2+. Nothing else.

## Versioning & Tags

- `version` (semver) in metadata is source of truth.
- On save: new version must be strictly greater than latest. Server rejects otherwise unless admin uses `--force`.
- Server creates lightweight git tag `<type>/<name>@<version>`.
- `versions()` lists tags matching prefix, sorted semver-desc.
- Install ref resolution:
  - `name` → latest tag
  - `name@1.2.0` → exact tag
  - `name@^1.0` → highest matching tag (semver range)
  - `name@HEAD` → tip of main (pre-release)
- Rollback from UI: pick old version → server creates a new commit restoring that tree, bumps version (e.g. `1.2.0 → 1.2.1`). Never rewrites history.
- Delete: commit removing the folder. Tags remain. Names are unique across the repository's lifetime (no reuse after delete).

## Search

- `IndexService` holds `List<ArtifactMeta>` in memory, rebuilt from `INDEX.yaml` on startup and after every commit.
- Lucene `RAMDirectory` for full-text over `title + description + content-head` and faceted filtering on `type/harness/tag/category/language/author`.
- Not persisted — rebuilt from memory model on startup. <10k artifacts expected; overhead negligible.
- No external search service.

## Docker Deployment

Multi-stage image. Build with Maven + Eclipse Temurin 21. Runtime base: `eclipse-temurin:21-jre-alpine` plus `git`, `bash`, `curl`, `jq`.

**Container layout:**

```
/app/app.jar
/app/scripts/gen-password-hash.sh
/data/library.git         # bare repo (volume)
/data/library-work/       # working checkout (volume)
/data/users.yaml          # (volume)
/data/config/             # app overrides (volume)
```

**docker-compose:**

```yaml
services:
  agent-library:
    image: agent-library:latest
    ports: ["8080:8080"]
    volumes: ["library-data:/data"]
    environment:
      SPRING_PROFILES_ACTIVE: prod
      LIBRARY_REPO_PATH: /data/library.git
      LIBRARY_WORK_PATH: /data/library-work
      LIBRARY_USERS_FILE: /data/users.yaml
    restart: unless-stopped
volumes:
  library-data:
```

**First-run bootstrap:** if `/data/library.git` is missing, `git init --bare` and seed with an empty `INDEX.yaml` plus the `agent-library-install` skill (self-hosting). Users file is created with a single admin account on first boot, printing the generated password once to the container log.

TLS termination handled by an external reverse proxy (Caddy/Traefik). Not bundled.

## Testing

- **Unit:** metadata schema validation, semver compare, install-manifest resolver, JSON merge.
- **Integration (Spring `@SpringBootTest` + tmp git dir):** full save → index → search → bundle → install-manifest.
- **Repo layer:** JGit against temp bare repo — commit races, tag creation, tree walking.
- **Web:** `MockMvc` for controllers + HTMX fragments (assert `hx-*` attributes in rendered HTML).
- **CLI:** `bats-core` against an ephemeral server container.
- **E2E:** Testcontainers spins up the built image; shell tests install a real skill and verify files on disk.

No mocked git. Real JGit, real tmp directories.
